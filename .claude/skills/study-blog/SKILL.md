# study-blog — 학습 내용을 블로그 글로 정리하기

프론트엔드 개발자가 백엔드를 공부하며 남기는 **블로그용 학습 노트**를 마크다운 파일로 만든다.
독자는 "나와 비슷한 프론트 개발자"라고 가정하고, 내가 나중에 다시 읽어도 이해되게 쓴다.

## 언제 쓰나

- "오늘 배운 거 블로그 글로 정리해줘", "학습 노트 만들어줘", "이번에 한 거 md로 써줘" 같은 요청
- `/study-blog` 로 직접 호출됐을 때
- 인자로 주제/범위를 주면(예: `/study-blog 오늘 한 라우팅`) 그 범위로 좁힌다. 없으면 **이번 세션에서 다룬 내용 전체**를 대상으로 한다.

## 작성 절차

1. **범위 파악**: 이번 대화에서 실제로 공부·실습한 내용을 추린다. 아직 안 배운 로드맵 뒷 단계 개념을 억지로 끌어오지 않는다.
2. **CLAUDE.md의 "진행 상황" 섹션을 참고**해 지금이 몇 단계인지 확인하고, 글 머리에 단계를 표기한다.
3. 실제 작성한 **코드/파일 경로/명령어/에러 로그**가 있으면 그대로 인용한다. 지어내지 않는다. 불확실하면 파일을 읽어 확인한다.
4. 아래 형식으로 마크다운을 작성한다.
5. `study-blog/` 폴더에 저장한다(없으면 만든다). 파일명: `YYYY-MM-DD-{짧은-영문-슬러그}.md` (예: `2026-07-03-nestjs-routing.md`).
6. 저장 후 파일 경로를 알려주고, 빠진 내용이나 톤 수정할 게 있는지 물어본다.

## 글의 형식 (이 구조를 지킨다)

```markdown
# {제목 — 그날 배운 핵심을 한 줄로}

> 백엔드 학습 기록 · 단계 {N} — {단계 이름}
> {YYYY-MM-DD}

## 들어가며
{오늘 뭘 하려 했는지, 왜 이걸 배웠는지 1~2문장}

## 배운 개념
{개념마다 소제목. 각 개념은 "무엇인지 → 왜 필요한지" 순서.
프론트(React/Next) 비유가 있으면 함께 적는다.}

## 실제로 해본 것
{작성한 코드 블록, 실행한 명령어, 접속한 URL과 결과.
코드는 실제 파일에서 가져온 것만.}

## 막혔던 점 / 삽질
{에러가 있었으면 에러 로그 → 원인 → 해결. 없으면 이 섹션 생략 가능.}

## 오늘의 정리
{3줄 이내 핵심 요약. 다음에 이어서 할 것 한 줄.}
```

## 톤 & 원칙

- **한국어**, 블로그 말투(반말/친근한 서술체 허용). 하지만 개념 설명은 정확하게.
- **"왜?"를 항상 포함**한다. 코드만 나열하지 말고 그 코드가 왜 그렇게 생겼는지 적는다.
- 프론트 개발자 관점의 **비유**를 적극 활용한다(컴포넌트·훅·상태관리·라우터 등).
- 실제로 안 한 걸 한 것처럼 쓰지 않는다. 세션에서 다룬 사실만 기록한다.
- 한 편이 너무 길면 주제별로 나눠 여러 파일로 저장할지 제안한다.
